🗺️ CSDM 5 – Kompleksowe podsumowanie

Co to jest CSDM?

  • Framework – NIE produkt, NIE licencja, NIE moduł do instalacji
  • Sposób organizacji i standaryzacji danych o usługach IT i biznesowych w ServiceNow
  • Wszystkie OOB tabele dostępne jako część platformy (ship z-boot lub free store apps)
  • NIE wymaga dodatkowych licencji ITOM Discovery
  • Elementy Foundation (Company, BU, Department, Location, User, Group) NIE są CI

3 podejścia implementacyjne

  • Outcome Centric – zacznij od wyników biznesowych, dobierz domeny
  • Domain Centric – skup się na konkretnej domenie (np. tylko Service Delivery)
  • Step-wise – stopniowe dojrzewanie wg etapów Crawl → Walk → Run → Fly

5 etapów implementacji (maturity stages)

  • FoundationCompany, BU, Department, Location, User, Group – podstawowe dane, Hardware CIs
  • CrawlBusiness Services + Technology Mgmt Services, podstawowe relacje CI↔Service
  • WalkService Instances (dawniej App Services), Service Offerings, lepsze mapowania
  • RunPełny model serwisowy, Dynamic CI Groups, Service Portfolios, compliance
  • FlyAutomatyzacja, AI/ML insights, pełna widoczność biznes ↔ IT

Operational CIs (używane w ITSM/IPC)

  • Business Service – jak klient widzi usługę
  • Technology Mgmt Service – jak IT zarządza usługą
  • Service Instance – instancja aplikacji na infrastrukturze

Grupy synchronizowane z Tech Mgmt Service Offering na CI

  • Managed by Group + Support Group + Change Group
  • Priorytet: Tech Mgmt Offering override CI Class Manager
  • NIE synchronizuje: Approval Group

⚠️ Na co uważać na egzaminie – CSDM 5

  • CSDM to NIE produkt / NIE SKU / NIE moduł – nie ma kodu do instalacji
  • Foundation: Company, Location, BU, User, Group NIE są CI – to dane referencyjne
  • Technology Management Service = dawniej Technical Service (zmiana w CSDM 5!)
  • Service Instance = dawniej Application Service (zmiana w CSDM 5!)
  • Grupy z Tech Mgmt Offering: Managed by + Support + Change (NIE Approval!)
  • CSDM NIE wymaga licencji ITOM – wszystkie OOB tabele są darmowe
  • Business Service = perspektywa klienta; Tech Mgmt Service = perspektywa IT
  • Etap to Foundation → Crawl → Walk → Run → Fly (5 etapów, nie 4)

🔧 CI Class Manager – kluczowe fakty

  • Centralne miejsce: atrybuty, IRE rules, suggested relationships, Basic Info
  • Principal Class → Basic Info tab → checkbox → filtruje CI reference fields na formularzach ITSM
  • Role do zarządzania: sn_cmdb_admin + personalize_dictionary
  • Pinned Classes = streamline zarządzania konkretną klasą
  • Nowa klasa custom: u_cmdb_ci_ prefix
  • Nową klasę zawsze dodawaj pod odpowiednią klasą nadrzędną
  • Najpierw sprawdź CMDB CI Class Models store app czy klasa już istnieje
  • Added tab = atrybuty stworzone dla konkretnej klasy (np. Windows Server)
  • Health Inclusion Rules → zawężają scope CIs w procesie identyfikacji (IRE)
  • Multisource property: glide.identification_engine.multisource_enabled

⚙️ IRE – Identification & Reconciliation Engine

  • Allow fallback to parent's rules = jeśli brak dopasowania, próbuje regułę klasy nadrzędnej
  • Class Upgrade = Server → Windows Server (bardziej szczegółowa klasa)
  • Class Downgrade = Windows Server → Server (mniej szczegółowa)
  • Class Switch = zmiana na inny branch hierarchii
  • Criterion attribute dla tej samej apki kilka razy na serwerze → Configuration File Path
  • Reconciliation priority: niższy numer = wyższy priorytet (100 > 200)
  • Wyższy priorytet może UPDATE niższy; niższy priorytet nie może update wyższy
  • Każde źródło może INSERT nowe rekordy niezależnie od priorytetu
  • Data Refresh Rules = pozwala niższemu źródłu update po X dniach bez zmian z wyższego

📥 Ingest – metody wgrywania danych

  • Service Graph Connector = fastest time to value, pre-built, minimizes tech debt
  • IntegrationHub ETL = recommended by IRE, field-to-field mapping, 3rd party & non-CMDB
  • Import Sets + Transform Maps = stara metoda; użyj CMDBTransformUtil API dla IRE
  • Agent Client Collector (ACC) = urządzenia w bezpiecznych środowiskach + intermittently connected
  • ServiceNow Discovery = automatyczne relacje; agentless; sieć LAN
  • Service Mapping = automatyczne relacje Application Service (top-down)
  • SGC customization → upgrade skips zmienione mappings (skipped change created)
  • Po customizacji SGC → customer odpowiada za support tych customizations
  • SGC Central tab dostępny w: CMDB Workspace
  • Import Sets + Transform Map: script type onBefore dla IRE

📊 CMDB Health Dashboard – 6 metryk

  • Correctness scorecard zawiera: Duplicate CIs, Orphan CIs, Stale CIs
  • Completeness scorecard: Required fields, Recommended fields (brakujące atrybuty)
  • Compliance scorecard: Audits (certyfikaty, patch level itp.)
  • Stale CIs = nie były update'owane → pole: sys_updated_on
  • Health Inclusion Rule = zawęża scope klasy w danej metryce (np. Orphan tylko 50K z 60K)
  • Health scores generowane przez: scheduled jobs (muszą być aktywne)
  • Compliance scorecard requires: Certification Filter + Certification Template + Audit

🗃️ CMDB Data Manager – polityki

  • Attestation = potwierdzenie że CI ISTNIEJE (ktoś potwierdza/odrzuca)
  • Certification = walidacja i UPDATE atrybutów CI (zadanie przypisane do osoby)
  • Archive Policy = zarządzanie wolumenem danych CMDB; usuwa stare
  • Delete Policy = po Retire → Delete (po potwierdzeniu)
  • Retire Policy = trigger lifecycle retirement
  • Prerequisite do archive/delete: CI musi być Retired + End of Life
  • Data Manager tasks w CMDB Workspace: Management tab
  • Data Manager generuje tasks w CMDB Workspace → My Work tab
  • De-dup tasks IRE → CMDB Workspace → Home → Important actions
  • Group reference w Data Manager policy: Managed By Group

🔄 CMDB 360 / Multisource

  • Enable: glide.identification_engine.multisource_enabled
  • Aktywuje Dynamic Reconciliation Rules
  • Dynamic Rule Types: Smallest Value, Most Reported, Largest Value
  • IRE reconciliation rules działają bez CMDB 360 (niezależnie od licencji ITOM Discovery)
  • Hierarchia: klasa bardziej szczegółowa (Windows Server) override klasa nadrzędna (Server)
  • Most Reported → wartość zgłoszona przez największą liczbę źródeł
  • Review field-to-field mappings SGC: IntegrationHub ETL

🗺️ CSDM – domeny i etapy

  • 7 domen: Foundation, Ideation & Strategy, Design & Planning, Build & Integration, Service Delivery, Service Consumption, Manage Portfolio
  • Foundation = Company, BU, Department, Location, User, Group (NIE są CI!)
  • Business Service = jak klient widzi serwis (cmdb_ci_service_business)
  • Technology Management Service = dawniej Technical Service (cmdb_ci_service_technical)
  • Service Instance = dawniej Application Service (cmdb_ci_service_auto)
  • Operational CIs (ITSM/IPC): Business Service, Tech Mgmt Service, Application Service
  • Etapy: Foundation → Crawl → Walk → Run → Fly
  • Podejścia: Outcome Centric, Domain Centric, Step-wise (Crawl/Walk/Run/Fly)
  • CSDM NIE jest produktem/SKU – to framework bez kodu do instalacji

💡 Insight – analiza i raportowanie

  • Natural Language Query (NLQ) = CMDB Search; agent-based; tworzenie w CMDB Workspace
  • CMDB Query Builder = complex queries wieloklasowe, CMDB 360 saved queries
  • Unified Map = widok zależności (Dependency View)
  • Playbooks = structured guidelines: Problem Overview → Problem Analysis → Importance → Fix/Improve
  • Playbooks dostępne w: CMDB Data Foundations Dashboard
  • Playbooks NIE automatycznie naprawiają – pomagają ANALIZOWAĆ i NAPRAWIAĆ
  • Business value CMDB: impact analysis, change risk, MTTR reduction

🔗 Asset-CI synchronizacja

  • Auto-sync: dwie business rules muszą być aktywne (CI→Asset + Asset→CI)
  • Asset-CI Field Mapping module = konfiguracja mapowania pól w obie strony
  • Asset State → CI: synchronizuje Install Status + Hardware Status (wg sample Q2)
  • Discovery source = SNAssetManagement dla CI stworzonych z assetu (bez discovery)
  • Last Discovered = empty → CI nie był odkryty przez Discovery
  • Dynamic CI Group na Change form → CI field → Affected CIs related list jest auto-populated

🏗️ Dynamic CI Groups i Technical Service Offerings

  • CMDB Group methods: Saved queries + Encoded queries (nie Tag-based, nie Scripted)
  • Groups syncowane na CI z Tech Mgmt Offering: Managed by Group + Support Group + Change Group
  • Priorytet dla Managed By Group: Technical Service Offering override CI Class Manager
  • Dynamic CI Group na Change form → Affected CIs related list auto-populated
  • Dynamic CI Group classification: Application Service LUB Technical Service (Service Classification)
  • CI nie powinien być w wielu Dynamic CI Groups powiązanych z różnymi Tech Mgmt Services
  • Support Group na Incident po wybraniu CI → pochodzi ze Assignment Group (nie Support Group)

🖥️ CMDB Workspace – zakładki i funkcje

  • Home tab → Important Actions (de-dup tasks, IRE issues), Health overview
  • My Work tab → Data Manager tasks przypisane do Ciebie (Attestation, Certification)
  • Management tab → Data Manager tasks dla menedżera (Attestation, Certification policies)
  • Health tab → CMDB Health Dashboard (Correctness, Completeness, Compliance)
  • SGC Central tab → zarządzanie Service Graph Connectors
  • De-dup tasks (IRE): Home → Important Actions
  • NLQ (Natural Language Query) tworzone i uruchamiane z CMDB Workspace
  • Playbooks dostępne w: CMDB Data Foundations Dashboard (nie Workspace!)

  • Discovery source dla CI z assetu: SNAssetManagement (bez discovery)
