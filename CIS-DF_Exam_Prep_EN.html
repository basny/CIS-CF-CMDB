🔧 CI Class Manager – key facts

  • Central location: attributes, IRE rules, suggested relationships, Basic Info
  • Principal Class → Basic Info tab → checkbox → filters CI reference fields on ITSM forms
  • Required roles: sn_cmdb_admin + personalize_dictionary
  • Pinned Classes = streamline management of a specific class
  • New custom class: u_cmdb_ci_ prefix
  • Always add a new class under the appropriate parent class
  • First check the CMDB CI Class Models store app to see if the class already exists
  • Added tab = attributes created for a specific class (e.g. Windows Server)
  • Health Inclusion Rules → narrow the scope of CIs in the IRE identification process
  • Multisource property: glide.identification_engine.multisource_enabled

⚙️ IRE – Identification & Reconciliation Engine

  • Allow fallback to parent's rules = if no match, tries the parent class rule
  • Class Upgrade = Server → Windows Server (more specific class)
  • Class Downgrade = Windows Server → Server (less specific)
  • Class Switch = change to a different branch of the hierarchy
  • Criterion attribute for same app installed multiple times on one server → Configuration File Path
  • Reconciliation priority: lower number = higher priority (100 > 200)
  • Higher priority can UPDATE lower; lower priority cannot update higher
  • Every source can INSERT new records regardless of priority
  • Data Refresh Rules = allows lower-priority source to update after X days without changes from higher-priority source

📥 Ingest – data loading methods

  • Service Graph Connector = fastest time to value, pre-built, minimizes tech debt
  • IntegrationHub ETL = recommended by IRE, field-to-field mapping, 3rd party & non-CMDB
  • Import Sets + Transform Maps = legacy method; use CMDBTransformUtil API for IRE
  • Agent Client Collector (ACC) = devices in secure environments + intermittently connected
  • ServiceNow Discovery = automatic relationships; agentless; LAN network
  • Service Mapping = automatic Application Service relationships (top-down)
  • SGC customization → upgrade skips changed mappings (skipped change created)
  • After SGC customization → customer is responsible for supporting those customizations
  • SGC Central tab available in: CMDB Workspace
  • Import Sets + Transform Map: script type onBefore for IRE

📊 CMDB Health Dashboard – 6 metrics

  • Correctness scorecard contains: Duplicate CIs, Orphan CIs, Stale CIs
  • Completeness scorecard: Required fields, Recommended fields (missing attributes)
  • Compliance scorecard: Audits (certificates, patch level, etc.)
  • Stale CIs = not updated → field: sys_updated_on
  • Health Inclusion Rule = narrows the scope of a class for a metric (e.g. Orphan only 50K of 60K)
  • Health scores generated by: scheduled jobs (must be active)
  • Compliance scorecard requires: Certification Filter + Certification Template + Audit

🗃️ CMDB Data Manager – policies

  • Attestation = confirmation that the CI EXISTS (someone confirms/rejects)
  • Certification = validation and UPDATE of CI attributes (task assigned to a person)
  • Archive Policy = manages CMDB data volume; removes old records
  • Delete Policy = after Retire → Delete (after confirmation)
  • Retire Policy = trigger lifecycle retirement
  • Prerequisite for archive/delete: CI must be Retired + End of Life
  • Data Manager tasks in CMDB Workspace: Management tab
  • Data Manager generates tasks in CMDB Workspace → My Work tab
  • IRE de-dup tasks → CMDB Workspace → Home → Important actions
  • Group reference in Data Manager policy: Managed By Group

🔄 CMDB 360 / Multisource

  • Enable: glide.identification_engine.multisource_enabled
  • Activates Dynamic Reconciliation Rules
  • Dynamic Rule Types: Smallest Value, Most Reported, Largest Value
  • IRE reconciliation rules work without CMDB 360 (independent of ITOM Discovery license)
  • Hierarchy: more specific class (Windows Server) overrides parent class (Server)
  • Most Reported → value reported by the largest number of sources
  • Review field-to-field mappings for SGC: IntegrationHub ETL

🗺️ CSDM – domains and stages

  • 7 domen: Foundation, Ideation & Strategy, Design & Planning, Build & Integration, Service Delivery, Service Consumption, Manage Portfolio
  • Foundation = Company, BU, Department, Location, User, Group (NOT CIs!)
  • Business Service = how the client sees the service (cmdb_ci_service_business)
  • Technology Management Service = formerly Technical Service (cmdb_ci_service_technical)
  • Service Instance = formerly Application Service (cmdb_ci_service_auto)
  • Operational CIs (ITSM/IPC): Business Service, Tech Mgmt Service, Application Service
  • Stages: Foundation → Crawl → Walk → Run → Fly
  • Approaches: Outcome Centric, Domain Centric, Step-wise (Crawl/Walk/Run/Fly)
  • CSDM is NOT a product/SKU – it is a framework with no code to install

💡 Insight – analytics and reporting

  • Natural Language Query (NLQ) = CMDB Search; agent-based; created in CMDB Workspace
  • CMDB Query Builder = complex multi-class queries, CMDB 360 saved queries
  • Unified Map = dependency view
  • Playbooks = structured guidelines: Problem Overview → Problem Analysis → Importance → Fix/Improve
  • Playbooks available in: CMDB Data Foundations Dashboard
  • Playbooks do NOT automatically fix – they help ANALYZE and REMEDIATE
  • Business value of CMDB: impact analysis, change risk, MTTR reduction

🔗 Asset-CI synchronization

  • Auto-sync: two business rules must be active (CI→Asset + Asset→CI)
  • Asset-CI Field Mapping module = configuration of field mapping in both directions
  • Asset State → CI: synchronizes Install Status + Hardware Status (per sample Q2)
  • Discovery source = SNAssetManagement for CIs created from an asset (without discovery)
  • Last Discovered = empty → CI was not discovered by Discovery
  • Dynamic CI Group on Change form → CI field → Affected CIs related list is auto-populated

🏗️ Dynamic CI Groups and Technical Service Offerings

  • CMDB Group methods: Saved queries + Encoded queries (not Tag-based, not Scripted)
  • Groups synced to CI from Tech Mgmt Offering: Managed by Group + Support Group + Change Group
  • Priority for Managed By Group: Technical Service Offering overrides CI Class Manager
  • Dynamic CI Group on Change form → Affected CIs related list auto-populated
  • Dynamic CI Group classification: Application Service OR Technical Service (Service Classification)
  • A CI should not be in multiple Dynamic CI Groups linked to different Tech Mgmt Services
  • Support Group on Incident after selecting CI → comes from Assignment Group (not Support Group)
